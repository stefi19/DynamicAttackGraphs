Dynamic Attack Graphs using Differential Dataflow

Incremental maintenance of a subset of MulVAL-style attack graph rules. Instead of recomputing the full attack graph after every patch, CVE, or firewall change, the prototype propagates signed differences through a recursive dataflow computation.

Motivation

Why Dynamic Attack Graphs?

Attack graphs model how an attacker can move from an initial position through reachable services and vulnerabilities toward a target. They are useful for reasoning about patches, firewall changes, and exposure paths, but real networks change constantly.

A full recomputation after every change is simple and defensible, but it wastes work when the affected region is small. This project investigates whether Differential Dataflow can maintain recursive attack graph facts incrementally for a MulVAL-style rule subset.

Background

MulVAL-Style Rules and Materialisation

Datalog intuition

Base facts describe hosts, services, firewall denials, attacker starting positions, and goals. Rules derive materialised facts such as code execution, owned machines, and reached goals. 

execCode(A, Dst, P) :-
  execCode(A, Src, _),
  hacl(Src, Dst, Svc),
  vulExists(Dst, _, Svc, P).

Incremental maintenance

The accurate baseline is full recomputation after applying the same update. Incremental maintenance is useful only when it produces exactly the same derived facts faster for the workload under test.

Differential Dataflow

Signed Differences Through Recursive Operators

Input facts map/filter join antijoin iterate derived facts

Differential Dataflow represents updates as records with logical time and signed multiplicity. Insertions contribute positive differences; deletions contribute negative differences. The rule implementation maps Datalog-style conjunctions to keyed joins, projections to maps, stratified firewall negation to antijoins, recursion to iterative dataflow, and duplicate removal to distinctness.

Correctness Validation

Every Incremental Result Must Match Recompute

incremental_result_after_update == full_recomputation_result_after_update

Full recomputation

Applies the same update to base facts, rebuilds the computation, and acts as the main oracle.

Naive fixpoint

Uses simple HashSet iteration for small scenarios and validates the rule semantics independently.

Randomized tests

Seeded tests generate small networks and update sequences, then compare engine outputs after each update.

Explainability

Selected Provenance Trees

The provenance layer reconstructs one valid proof tree after computation. It is separate from Differential Dataflow internals and currently targets selected derived facts rather than enumerating all minimal attack paths. 

goalReached(eve, admin01)
└── ownsMachine(eve, admin01)
    └── execCode(eve, admin01, root)
        ├── execCode(eve, db01, root)
        ├── effectiveAccess(db01, admin01, smb)
        └── vulExists(admin01, CVE-2024-8888, smb, root)

Limitations

Current Research Boundaries

  • Subset of MulVAL-style rules.
  • Simplified CVE and privilege semantics.
  • Synthetic or semi-synthetic benchmark topologies.
  • Single-machine research prototype setup.
  • Provenance reconstructs selected explanations after computation.
  • No live vulnerability scanner integration yet.
